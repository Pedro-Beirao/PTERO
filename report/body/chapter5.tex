\section{Runtime Environment}
Out of all \acp{RTE} compatible with \textit{IEC 61499}, this prototype will focus on supporting \textit{DINASORE} (Dynamic INtelligent Architecture for Software and MOdular REconfiguration) \cite{pereira_dinasore_2020}. It is much less complex than others, making it ideal for a proof of concept \ac{IDE}. As an open-source project, \textit{DINASORE} can be freely inspected and studied, which is particularly valuable in a research and prototyping context where flexibility is prioritised over real-time performance guarantees. This ease of study rules out the use of commercial \acp{RTE} like \textit{EcoStruxure Automation Expert's "dPAC"} and \textit{ISaGRAF Runtime}. Other open-source \acp{RTE} were considered, but none were as well-suited as \textit{DINASORE}. \textit{4diac FORTE} for example was way too extensive and complicated to be able to easily study its source code. \Cref{fig:dinasore-architecture} represents the architecture of this \ac{RTE} where \textit{PTERO} aims to replace the \textit{4DIAC Interface}.

\section{Server}
The design begins by setting up a server that will serve the \ac{IDE} interface as a webpage, store and manage all the data and handle communication with the \acp{RTE}. Taking into account the features we plan on implementing, \textit{JavaScript} running on \textit{Node.js} were chosen due to the extensive library ecosystem. For real-time collaboration we will need a library to run in both the server and the frontend, since \textit{JavaScript} runs natively in the browser, it further reinforces that it should also be used in the server. The library choice will be explained in depth in \Cref{sec:real-time-collaboration}, proving the previous train of thought.

\subsection{Devices Configuration View}
In the interest of distributed \ac{CPPS}, \textit{IEC 61499} allows function blocks to be mapped to different \textit{DINASOREs}, so we need a clean way to define the resources that are available. Both the \textit{DINASORE} port and the \textit{SSH} port need to be defined for each defined since deployment communication will be made through both.
% TODO need to mention sync
% TODO something more

Each device should have several fields that the team can configure:

\begin{itemize}
  \item \textbf{Name}: Human friendly name to quickly identify what device this is
  \item \textbf{Address}: The address of the device
  \item \textbf{DINASORE port}: The port where \textit{DINASORE} is running
  \item \textbf{Color}: The color associated with this \textit{DINASORE} instance, providing a visual distinction in the \textit{Graph View} between function blocks mapped to different instances.
  \item \textbf{SSH port}: The open \textit{SSH} port of the device.
  \item \textbf{SSH user} and \textbf{SSH password}: The \textit{SSH} login credentials.
  \item \textbf{SSH path to DINASORE}: Path in the \textit{SSH} filesystem pointing to where \textit{DINASORE} is located.
\end{itemize}

\subsection{Function Blocks Editor View}
This is where individual function blocks are defined. It allows the team to specify the interface of each function block, its input and output data ports and events, as well as program the internal behaviour. This separation of interface and implementation follows the \textit{IEC 61499} model of encapsulation, where a function block exposes a well defined interface while keeping its internal logic self-contained.

Like in the other \acp{RTE}, in \textit{DINASORE} the interface of the function block is defined in a \textit{xml} format file with a \textit{.fbt} extension, which results in a function block that can be displayed graphically like seen in \Cref{fig:example-fb}. The internal behaviour of the function blocks in \textit{DINASORE} is programmed in \textit{Python}.

\subsection{Graph View}
This view is the primary workspace for composing the application. Function blocks are placed on the canvas and connected together by drawing connections between their data and event ports, forming the application network. Event ports are commonly color coded red, while data ports are color coded blue. % TODO talk about coloring

In \textit{DINASORE}, all \textit{INIT} event ports are by default assigned to \textit{EMB\_RES}, the default resource that sends an event when the application is started. This removes the need to connect the \textit{INIT} event ports, but the \textit{READ} event port is not assigned by default and therefore needs to be connected in every function block. The \texttt{schedule()} function will only be called and generate output when an event is sent to the \textit{READ} event port. The described event driven execution model is what allows an explicit specification of the execution order of function blocks. Periodic out of event execution of function blocks is also possible with the implementation of a looping function block that triggers events \cite{noauthor_iec_nodate}.

A function block will have some input and output data ports, each with its own associated data type. For example, an \textit{INT} input port will not accept data from a \textit{WORD} output port, and therefore should not be able to connect to it. It is also possible to set a default value for an input if no output is connected to it, there is an editable string field for this which gets converted to the expected data type by the \ac{RTE}. The data types supported by \textit{IEC 61499} are derived from and defined by \textit{IEC 61131-3} as listed in \Cref{tab:iec-data-types}.

This system was achieved in the prototype with the use of the \textit{LiteGraph.js} library, a graph node editor for the browser. A lot of other libraries were considered, but this one was chosen due to the requirements of the proposed solution. Let us discuss the alternatives that were considered to understand why this choice is important:

% TODO add references for each
\begin{itemize}
  \item \textbf{Rete.js}: A very flexible and complete editor framework, but the added complexity makes it harder to integrate the desired features. It is also a more general purpose editor, meaning the multi-input and multi-output function blocks with specific data types are not as well supported.
  \item \textbf{xyflow}: A highly customizable framework composed of React Flow and Svelte Flow. While it provides excellent visual customisation and integration with frontend frameworks, it suffers from the same problems as \textit{Rete.js}. % TODO any word better than problems?
  \item \textbf{Drawflow}: A lightweight and simplistic library with a focus on data flow execution. Although simple to integrate, it lacks many features required for designing \textit{IEC 61499} function blocks, such as robust type handling and multi-input and multi-output.
  \item \textbf{Baklava.js}: With a clean architecture and flexibility, this would be a good alternative, but since it specifically targets \textit{Vue} and requires a more intricate setup, it was not the most appropriate choice for a prototype.
  \item \textbf{GoJS}: Very powerful commercial alternative. It offers advanced functionality and flexibility, but its licensing model make it less suitable for lightweight research prototypes and open development environments.
  \item \textbf{Node-RED}: A flow based programming environment. While conceptually similar, its architecture is more focused on runtime execution than on being a visual editor of structured data.
\end{itemize}

Among these alternatives, \textit{LiteGraph.js} presented the best balance between simplicity, flexibility and ease of integration. Its builtin support for typed inputs and outputs, lightweight architecture and direct focus on visual data flow programming made it especially suitable for the requirements of the proposed solution. Even so, it was necessary to implement custom behaviour on top of some functions of \textit{LiteGraph.js} to achieve the desired behaviour. The choice of the graph library is very important, since the requirements of \textit{IEC 61499} differ significantly from those of most conventional graph editors, as such, special care must be taken to ensure that these requirements are properly supported.

\section{Real-Time Collaboration} \label{sec:real-time-collaboration}
In \textit{PTERO} the collaborative editing system was implemented using \textit{Yjs} \cite{cimiano_yjs_2015}, a \ac{CRDT} framework designed for distributed collaborative applications, allowing users to concurrently edit shared data structures while automatically resolving conflicts in a deterministic manner. \textit{Yjs} was chosen in detriment of alternative \ac{CRDT} implementations like \textit{Automerge} and \textit{Loro} due to its large ecosystem and simplicity. With the built-in support for many code editor libraries, adding real-time collaboration to the \textit{Function Blocks Editor View} was very straightforward.

In contrast, there was no built-in support for \textit{LiteGraph.js}, so the graph structure representing the \ac{FBD} application is synchronised through shared \textit{Yjs} data structures. Operations such as creating function blocks, deleting connections, modifying inputs or moving nodes are propagated in real time to all connected clients. Each user therefore observes the same graph state with minimal delay. To integrate collaboration into this editor, the internal state changes of the graph were connected to the shared \textit{Yjs} document. Whenever a local modification occurs in the graph, the corresponding operation is serialized and applied to the shared collaborative state. Likewise, remote updates received from other users are reflected back into the local graph editor instance.

\section{Deployment} \label{sec:deployment}
The deployment process is responsible for taking the application defined in the \textit{Graph View} and bringing it to life across the configured \textit{DINASORE} instances. It is broken down into a sequence of steps, each of which is described in the following subsections.

\subsection{Synchronise with DINASOREs}
The first step is to ensure that each \textit{DINASORE} instance has access to the function blocks that it will need to run the application. This is done by sending all the \textit{.fbt} and corresponding \textit{.py} files in the database, to every device. To achieve this we can use the \texttt{synchronize.py} script that accompanies the \textit{DINASORE} source code. A configuration file needs to be provided that tells the script where to send the function blocks to, like shown in \Cref{lst:sync-config}. The server must have a valid \textit{SSH} connection with each device in order to connect and send the files.

% First, \texttt{master-fbs-path} dictates where in the server filesystem the function block files that will be sent are located, then an array of \texttt{dinasores} with the configurations defined in the \textit{Devices Configuration View}. The \texttt{port} used here will be the \textit{SSH} port, the \textit{DINASORE} must have an open \textit{SSH} channel that the server is able to connect to via this port. This file is generated from the configured fields in the \textit{Devices Configuration View}. When the script is ran, it will read the contents of this file in order to connect and send the files to each \textit{DINASORE}.

\subsection{Send Commands}
Once the function block files are in place, the server will send the necessary commands to each \textit{DINASORE} instance to build and start the application. This includes creating the function block instances, writing the initial input values, establishing the connections between event and data ports and finally sending the start command to begin execution. The order in which these messages are issued is important, as connections can only be established between existing nodes. The following two subsections describe how the messages are constructed and what they contain.

\subsubsection{Build messages}
Before the messages can be sent, they need to be built into the format that \textit{DINASORE} is expecting. It consists of a two part message: the first part refers to the configuration name, which by default we will use \textit{EMB\_RES}, while the second is the actual message payload containing the wanted command. Each of these parts are preceded by a 3-byte header composed of a \texttt{0x50} byte followed by 2 bytes indicating the size of the corresponding content. A representation of this format is presented in \Cref{fig:build-message}.

\subsection{Console}
To give the team visibility into the deployment process right from the \ac{IDE}, without having to log into the server, a console is provided in the \textit{Console View} that displays a live log of each action taken during deployment. There are two main things being logged:

\begin{itemize}
  \item \textbf{Synchronise}: The output of running the \textit{Python} script is displayed so that the team see that the \textit{SSH} communication is being done properly.
  \item \textbf{Send Commands}: As commands are sent and responses are received, they are displayed, allowing the team to make sure each command is formulated correctly, being processed and a response is issued.
\end{itemize}

\subsection{Watch}
Once the application is running, the values of the node outputs are continuously retrieved from the \textit{DINASORE} instances and displayed in the \textit{Graph View}, providing immediate feedback on the state of the running application. This is particularly useful for debugging and validating the behaviour of the application during development. In order to get these values, the \ac{IDE} must tell the \textit{DINASORE} instances which output slots it wants to watch and then request the values. In this prototype, for simplicity, all data outputs are watched and we need to send a watch creation request for each.

\Cref{fig:sequence-watch} represents the sequence that the messages need to be sent. Details about these requests and the associated responses can be found in \Cref{app:watch}.
