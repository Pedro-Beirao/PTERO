\chapter{Validation} \label{chap:validation}
The purpose of this chapter is to evaluate whether the proposed solution achieves the proposed objectives, effectively addressing the gap previously identified. The evaluation combines objective measurements of system behaviour with subjective feedback collected from users. Together, these perspectives provide a comprehensive assessment of the prototype's effectiveness as an \textit{IEC 61499} development environment. The following questions are to be answered by this evaluation:

\begin{enumerate}
\item Can the proposed solution successfully support the complete development workflow of an \textit{IEC 61499} application?
\item Does the proposed architecture provide advantages over existing desktop based development environments?
\item Does the integration of real-time collaboration improve collaborative development workflows?
\item Do users perceive the proposed solution as accessible, easy to use and an improvement over previous solutions?
\end{enumerate}

To address these questions, the evaluation combined both quantitative and qualitative methods, assessing objective performance characteristics alongside subjective user experience. Five complementary areas make up this validation. Functional validation verifies that the prototype correctly supports the modelling, deployment and monitoring of \textit{IEC 61499} applications. Performance evaluation analyses workflow efficiency and compares key tasks against existing tools. Collaboration evaluation examines the benefits of the real-time collaborative architecture and contrasts it with traditional version control workflows. Usability evaluation focuses on user perception and experience, including a study of existing \textit{IEC 61499} tooling. Finally, the results are discussed in the context of the research objectives and identified limitations.

\section{Experimental Setup} \label{sec:experimental-setup}
A common test application was used throughout the validation process. The same application was implemented using both \textit{PTERO} and \textit{4DIAC-IDE}, allowing direct comparison between environments.

\subsection{Hardware}
The following hardware devices were available and used throughout the evaluation phase of this work:

\begin{enumerate}
\item \textbf{M1 MacBook Pro} (16 GB RAM, 2020, macOS 26): This machine served as the primary development and testing platform for \textit{PTERO}. Being the development machine, it hosted the local development server during iterative testing cycles and was used to evaluate the responsiveness of the collaborative editing interface, the correctness of graph synchronization and the performance of the \textit{Yjs} based real-time collaboration layer under local conditions.

\item \textbf{ASUS TUF Gaming F16} (32 GB RAM, 2024, Windows 11): This machine was employed primarily for evaluating the performance of \textit{4DIAC-IDE}.

\item \textbf{Raspberry Pi 3} (1 GB RAM, Raspberry Pi OS): This device acted as a permanently running, remotely accessible server throughout the evaluation period. It hosted a persistent instance of the \textit{PTERO} server as well as an instance of \textit{DINASORE}. Its limited memory and processing capacity made it the most representative of real-world target environments and its inclusion in the testing structure was instrumental in establishing the practical boundaries of the collaboration server.
\end{enumerate}

\subsection{Available Resources}
The following resources were made available in the testing environments:

\begin{itemize}
\item One \textit{DINASORE} \ac{RTE}.
\item Three function block definitions obtained from the official \textit{DINASORE} tutorial \textit{'Sensorization "Hello World!"'} \cite{noauthor_digi2-feupdinasore_2026}:
  \begin{itemize}

\end{itemize}

The selected function blocks provide a representative example of a small \textit{IEC 61499} application while remaining sufficiently simple for controlled testing.

Besides verifying that the output values are being monitored correctly, following the \textit{Deployed} step, the commands displayed in the console were directly compared between \acp{IDE}. It was verified that both were sending exactly the same commands, the headers as well as the contents were exactly the same and the responses were no different. While the two workflows are broadly equivalent in the number of steps required, they differ in several procedural details. Configuring a device in \textit{PTERO} requires additional SSH credentials upfront, a step absent in \textit{4DIAC-IDE}, however, this overhead is offset later in the workflow, as file synchronization with \textit{DINASORE} is a manual command line operation in \textit{4DIAC-IDE}, while it is handled automatically by \textit{PTERO} at deploy time. These differences reflect a deliberate design philosophy in \textit{PTERO} of consolidating configuration complexity into the tool itself.

\section{Performance Evaluation}
This solution not only aims to execute the same core workloads as \textit{4DIAC-IDE}, but also introduces architectural capabilities and design choices that justify its development. To assess its efficacy, the system's performance is evaluated against four primary metrics:

\begin{itemize}
    \item \textbf{Installation Complexity}: The technical overhead and environmental constraints required to set up the development environment.
    \item \textbf{Startup Latency}: The temporal duration from initialization to full operational readiness.
    \item \textbf{Configuration Overhead}: The number of manual steps required to configure the system.
    \item \textbf{Deployment Flexibility}: The flexibility that the deployment system allows.
\end{itemize}

The \textbf{Install} and \textbf{Open} tasks evaluated installation complexity and startup latency. While \textit{4DIAC-IDE} offers a straightforward local desktop installer, \textit{PTERO} trades this for a one-time backend setup that guarantees universal cross-platform access via standard web browsers. Furthermore, by offloading computationally heavy operations to the backend, \textit{PTERO} reduced client-side startup latency from 4 seconds down to under 1 second, fulfilling the objective for immediate operational readiness.

The remaining tasks evaluated configuration overhead and deployment flexibility. \textit{4DIAC-IDE} introduced operational friction by requiring external text editors for script writing, manual script execution for synchronization and asynchronous version control operations. It also requires complex network setups since every engineering workstation must maintain individual direct connections to each target \ac{RTE}. Conversely, \textit{PTERO} minimizes configuration overhead by unifying file editing, automating file synchronization and introducing the concept a \ac{CRDT}. By utilizing the server as a middleware, \textit{PTERO} dramatically improved deployment flexibility, as clients only need to authenticate with the centralized server, not the \acp{RTE}.

\section{Collaboration Evaluation}
\Citeauthor{cimiano_yjs_2015} \cite{cimiano_yjs_2015} establish that \textit{Yjs} is able to handle an average of 100 actions per millisecond which is obviously good enough for this use case, as it does not require millisecond precision. Taking connection with the remote server into account, a test was devised to understand how well the \textit{Raspberry Pi} could handle multiple users at the same time. A script was created that initialized a set number of clients, connected them to the remote server and sent a change that would get propagated across the clients. By measuring the time that would take for the clients to receive the change, the chart in \Cref{fig:yjs-performance} was obtained. The results demonstrated that even a resource constrained machine accessed over a network can serve as an adequate collaboration server for a small team, however, the \textit{Raspberry Pi} deployment was found to be unsuitable beyond approximately 240 simultaneous clients, at which point the hardware prevented additional connections from being established reliably.

\begin{figure}[H]
  \includegraphics[width=0.8\linewidth]{yjs-performance.png}
  \caption{Performance results of \textit{Yjs} running in the \textit{Raspberry Pi}.}
  \label{fig:yjs-performance}
\end{figure}

The same test was performed with a server running on the \textit{M1 MacBook Pro} and with the clients also in the same machine, removing any delay resulting from network latency and in a much more powerful machine. It was possible to connect up to 3000 clients with an average propagation time of 65 ms and a P95 of 221 ms, proving that if there is a need for hundreds of clients to be connected at once, the solution is able to scale with the hardware to accommodate it.

\section{Usability Evaluation}
Two questionnaires were designed and given to participants. One focused on the user experience of working with \textit{4DIAC-IDE} as the underlying development environment for \textit{IEC 61499}. The other asked both for a usability opinion on \textit{PTERO}, as well as a direct comparison between these two \acp{IDE}. The participants of the questionnaires were:

\begin{itemize}
\item 11 Students from \ac{MEEC} at \ac{FEUP}, enrolled in the course of \ac{SINF}, in which, over the course of a semester, they were tasked with using \textit{4DIAC-IDE} for modelling \ac{FBD} programs with \textit{DINASORE} as the \ac{RTE}. Having completed a full semester of work on this environment, these students possessed sufficient practical familiarity with the tool to provide informed usability assessments and represent a realistic profile of future \textit{PTERO} adopters in academic and educational contexts.
\item 5 Students at \ac{FEUP}, working on Master's dissertations at \ac{SYSTEC} on topics directly related to \textit{4DIAC-IDE} and \textit{DINASORE}. Their deeper technical engagement with the ecosystem made them particularly well positioned to evaluate the correctness and completeness of \textit{PTERO}'s workflow.
\end{itemize}

\subsubsection*{System Usability Scale}
To obtain a globally standardized and quantifiable metric of \textit{PTERO}'s overall user experience, the study incorporated the \ac{SUS} \cite{lewis_system_2018}. The \ac{SUS} is a reliable scale where the raw point responses are aggregated and a score from 0 to 100 is calculated.

\textit{PTERO} achieved an average SUS score of \textbf{88.75}. According to empirical usability benchmarks  \cite{lewis_item_2018}, this score positions \textit{PTERO} well above the 95th percentile of all analysed applications. This outstanding evaluation confirms that \textit{PTERO} is an extremely user-friendly solution.

\section{Discussion}
The evaluation results indicate that the proposed solution successfully achieves functional parity with the established \textit{4DIAC-IDE} while introducing architectural and usability improvements that address several of the limitations identified in traditional \textit{IEC 61499} development environments.

The functional validation confirmed that \textit{PTERO} correctly supports the complete development workflow, including configuration, modelling, deployment and monitoring of \textit{IEC 61499} applications. Both systems generated identical deployment commands and produced equivalent runtime behaviour in \textit{DINASORE}. This equivalence is an important result, as it demonstrates that the proposed abstraction layer does not compromise correctness or compatibility with existing tooling. Instead, it preserves interoperability while offering a different interaction model.

The performance evaluation further complements these findings by highlighting how the architectural decisions in \textit{PTERO} translate into measurable workflow improvements. While both systems are capable of executing equivalent \textit{IEC 61499} applications, \textit{PTERO} reduces the number of manual steps required across key development tasks such as project setup, synchronization and deployment. A central factor behind this improvement is the tighter integration between the different components of the development pipeline. In \textit{4DIAC-IDE}, the development workflow is distributed across multiple loosely connected tools and processes, requiring explicit user actions to maintain consistency between the modelling environment, local file system and remote runtime. In contrast, \textit{PTERO} internalizes these responsibilities within the platform itself, allowing project state, file management and deployment logic to be handled automatically as part of a single unified system.

The usability questionnaires provide strong evidence that a web-based implementation can offer a significantly improved user experience compared to \textit{4DIAC-IDE}. Participants consistently reported higher levels of perceived ease of use, intuitiveness and overall satisfaction when interacting with \textit{PTERO}. This improvement can be attributed not only to interface design choices, but also to the inherent advantages of web technologies in this context.

Furthermore, it was observed that the integration of all development tasks into a single unified interface contributes to a reduction in cognitive load. Unlike \textit{4DIAC-IDE}, where users must switch between multiple tools and external processes (such as file system management and synchronization scripts), the web-based environment consolidates modelling, editing, deployment and monitoring into a coherent workflow. The results also suggest that responsiveness and perceived simplicity play an important role in user satisfaction.

Overall, the findings indicate that the adoption of web technologies is not merely a change in deployment platform, but a design decision that directly influences usability by simplifying access, unifying workflows and reducing structural complexity for end users.
