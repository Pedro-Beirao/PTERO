\chapter{Validation}
The purpose of this chapter is to evaluate whether the proposed solution achieves the proposed objectives, effectively addressing the gap previously identified. The evaluation combines objective measurements of system behaviour with subjective feedback collected from users. Together, these perspectives provide a comprehensive assessment of the prototype's effectiveness as an \textit{IEC 61499} development environment.

The evaluation is organised into five complementary areas. Functional validation verifies that the prototype correctly supports the modelling, deployment and monitoring of \textit{IEC 61499} applications. Performance evaluation analyses workflow efficiency and compares key tasks against existing tools. Collaboration evaluation examines the benefits of the real-time collaborative architecture and contrasts it with traditional version control workflows. Usability evaluation focuses on user perception and experience, including a study of existing \textit{IEC 61499} tooling. Finally, the results are discussed in the context of the research objectives and identified limitations.

\section{Evaluation Goals}
The evaluation aims to answer the following questions:

\begin{enumerate}
\item Can the proposed solution successfully support the complete development workflow of an \textit{IEC 61499} application?
\item Does the proposed architecture provide advantages over existing desktop based development environments?
\item Does the integration of real-time collaboration improve collaborative development workflows?
\item Do users perceive the proposed solution as accessible, easy to use and an improvement over previous solutions?
\end{enumerate}

To answer these questions, both quantitative and qualitative evaluation methods were employed.

\section{Experimental Setup}
A common test application was used throughout the validation process. The same application was implemented using both the developed prototype and \textit{4diac IDE}, allowing direct comparison between environments.

\subsection{Hardware}
The following hardware devices were available and used throughout the evaluation phase of this work:

\begin{enumerate}
\item \textbf{M1 MacBook Pro} (16GB RAM, 2020): This machine served as the primary development and testing platform for \textit{PTERO}. Being the development machine, it hosted the local development server during iterative testing cycles and was used to evaluate the responsiveness of the collaborative editing interface, the correctness of graph synchronisation and the performance of the \textit{Yjs} based real-time collaboration layer under local conditions.

\item \textbf{ASUS TUF Gaming F16} (32GB RAM, 2024): This machine was employed primarily for evaluating the performance of the \textit{4diac} runtime environment.

\item \textbf{Raspberry Pi 3} (1GB RAM): This device computer acted as a permanently running, remotely accessible server throughout the evaluation period. It hosted a persistent instance of the \textit{PTERO} server as well as an instance of \textit{DINASORE}. Its limited memory and processing capacity made it the most representative of real-world target environments and its inclusion in the testing structure was instrumental in establishing the practical boundaries of the collaboration server.
\end{enumerate}

\section{Functional Validation}
\textit{PTERO} was subjected to functional tests to confirm that all core features behave as specified. It was tested by implementing and deploying the test program defined in \Cref{sec:test-program-design}. As a control, the same was performed in \textit{4diac IDE}. The steps taken are as follows:

% TODO which hardware

\begin{enumerate}
\item \textbf{Configured available \textit{DINASORE}}
  \begin{itemize}
  \item \textbf{4diac IDE}: In the \textit{System Configuration} tab, added a device and set its IP address and the port where \textit{DINASORE} is running, then connected it to the \textit{Ethernet} tunnel.
  \item \textbf{PTERO}: In the \textit{Config} tab, added a device and set its IP address and the port where \textit{DINASORE} is running, as well as the SSH details of the device which will be used to sync the function blocks files.
  \end{itemize}

\item \textbf{Imported the function block files}
  \begin{itemize}
  \item \textbf{4diac IDE}: Copied and pasted the given files to \texttt{4diac-ide/typelibrary/(new\_folder)}.
  \item \textbf{PTERO}: Copied and pasted the given files in the \textit{Function Blocks} tab.
  \end{itemize}

\item \textbf{Designed the flow}
  \begin{itemize}
  \item \textbf{4diac IDE}: In the \textit{App View} tab, dragged and dropped the wanted function blocks, then connected the event and data slots. Double-clicked input slots to assign input value.
  \item \textbf{PTERO}: In the \textit{Graph} tab, right-clicked and selected the wanted function blocks to add them, then connected the event and data slots. Right-clicked nodes to assign input value.
  \end{itemize}

\item \textbf{Mapped to \textit{DINASORE}}
  \begin{itemize}
  \item \textbf{4diac IDE}: Right-clicked nodes to map them to the device. They became color coded to show which device they were mapped to.
  \item \textbf{PTERO}: Right-clicked nodes to map them to the device. They became color coded to show which device they were mapped to.
  \end{itemize}

\item \textbf{Synced files with \textit{DINASORE}}
  \begin{itemize}
  \item \textbf{4diac IDE}: Called the \texttt{synchronize.py} manually, feeding it the needed SSH details.
  \item \textbf{PTERO}: This was done automatically and without any needed setup when deploying.
  \end{itemize}

\item \textbf{Deployed}
  \begin{itemize}
  \item \textbf{4diac IDE}: Pressed the \textit{Deploy} button. The sent commands and associated responses were displayed in a console.
  \item \textbf{PTERO}: Pressed the \textit{Deploy} button. The sent commands and associated responses were displayed in a console.
  \end{itemize}

\item \textbf{Monitored the output}
  \begin{itemize}
  \item \textbf{4diac IDE}: The \textit{Monitor values} option was enabled, and the values were displayed on the right side of the output data slots.
  \item \textbf{PTERO}: The values were displayed on the right side of the output data slots.
  \end{itemize}
\end{enumerate}

Besides verifying that the output values are being monitored correctly, following the \textit{Deployed} step, the commands displayed in the console were directly compared between \acp{IDE}. It was verified that both were sending exactly the same commands, the headers as well as the contents were exactly the same, and the responses were no different. These results confirmed that the prototype correctly handles the development of \textit{IEC 61499} programs.

\section{Performance Evaluation}
This solution not only aims to be able to do the same workload as \textit{4diac IDE}, but also have capabilities and an inherent design that justify its existence. Therefore, this section compares the performance of several real world tasks between the \acp{IDE}. The listed tasks are the ones that demonstrated to be significantly different between solutions, others were analysed but proved to not be worth the discussion.

\begin{itemize}
\item \textbf{Install}
  \begin{itemize}
  \item \textbf{4diac IDE}: Install a desktop app. There is no compatible version for macOS.
  \item \textbf{PTERO}: Install a \textit{Node.js} server. By working in a web browser, it is crossplatform without much development effort, which allows macOS users to use it. The downside is that it is harder to install for just local development.
  \end{itemize}

\item \textbf{Open}
  \begin{itemize}
  \item \textbf{4diac IDE}: Average of 4 seconds to boot the application in a mid level computer. As it is written in \textit{Java}, the boot time cannot be improved much.
  \item \textbf{PTERO}: Average of less than a second to load webpage. The use of lightweight libraries and the move of heavy workloads to the server allow the page to load in an instant.
  \end{itemize}

\item \textbf{Create and edit function block files}
  \begin{itemize}
  \item \textbf{4diac IDE}: While it does have a way to create and edit the \textit{.fbt} files inside the editor, the code editor is not targeted against \textit{DINASORE} and therefore does not support \textit{.py} files.
  \item \textbf{PTERO}: Has a text editor for both types of files.
  \end{itemize}

\item \textbf{Collaboration}
  \begin{itemize}
  \item \textbf{4diac IDE}: An external periodic version control system like \textit{Git} needs to be used. Structured data in formats like \textit{XML} is not handled properly by these types of version control system, which results in frequent merge conflicts \cite{axelsson_version_2015}. The following test was conducted:
    \begin{enumerate}
    \item A project has a graph with 1 function block added. 2 users are working on the graph.
    \item User \#1 deletes the function block.
    \item User \#2 modifies the position of the function block.
    \item User \#1 commits and pushes.
    \item User \#2 tries to pull the changes but gets a merge conflict
    \end{enumerate}
  \item \textbf{PTERO}: Has a realtime collaboration system. Any change made is immediately propagated to every client, and resolved by a \ac{CRDT}, ensuring no conflicts need to be handled manually and damage is minimised.
  \end{itemize}

\item \textbf{Synchronising files with the \textit{DINASOREs}}
  \begin{itemize}
  \item \textbf{4diac IDE}: Required to manually use an external script before deployment.
  \item \textbf{PTERO}: The server itself will connect and synchronise the files with the \textit{DINASOREs}. This happens automatically when attempting to deploy, ensuring that deployments never fail due to the human error of forgetting to sync the files.
  \end{itemize}

\item \textbf{Remote deployment}
  \begin{itemize}
  \item \textbf{4diac IDE}: Every \ac{IDE} must have a valid connection with each available \textit{DINASORE}, as well as \textit{SSH} access. This gets increasingly complicated if the user want to make a deployment while in a different network. A way to expose each individual device needs to be design.
  \item \textbf{PTERO}: By having the server as the middleware, only it must be exposed for the users to connect. With only the server permanently having direct connection with the \textit{DINASORE} devices, no individual handling is needed.
  \end{itemize}
\end{itemize}

\Citeauthor{cimiano_yjs_2015} claims that \textit{Yjs} is able to handle an average of 100 actions per millisecond \cite{cimiano_yjs_2015} which is obviously good enough for this use case, as it does not require millisecond precision. Taking connection with the remote server into account, a test was devised to understand how well the \textit{RaspberryPi} could handle multiple users at the same time. A script was created that initialised a set number of clients, connected them to the remote server and sent a change that would get propagated across the clients. By measuring the time that would take for the client to receive the change, the chart in \Cref{fig:yjs-performance} was obtained. The results demonstrated that even a resource constrained machine accessed over a network can serve as an adequate collaboration server for a small team, however, the \textit{Raspberry Pi} deployment was found to be unsuitable beyond approximately 240 simultaneous clients, at which point the hardware prevented additional connections from being established reliably.

\begin{figure}[H]
  \includegraphics[width=0.8\linewidth]{yjs-performance.png}
  \caption{Performance results of \textit{Yjs} running in the \textit{RaspberryPi}}
  \label{fig:yjs-performance}
\end{figure}

The same test was performed with a server running on the \textit{M1 MacBook Pro} and the clients also in the same machine, removing any delay resulting from network delay and in a much more powerful machine. It was possible to connect 3000 clients with an average propagation time of 65 ms and a P95 of 221 ms, proving that if there is a need for hundreds of clients to be connected at once, the solution is able to scale with the hardware to accommodate.

\section{Discussion}
The evaluation results indicate that the proposed solution successfully achieves functional parity with the established \textit{4diac IDE} while introducing architectural and usability improvements that address several of the limitations identified in traditional \textit{IEC 61499} development environments.

The functional validation confirmed that \textit{PTERO} correctly supports the complete development workflow, including configuration, modelling, deployment and monitoring of \textit{IEC 61499} applications. Both systems generated identical deployment commands and produced equivalent runtime behaviour in \textit{DINASORE}. This equivalence is an important result, as it demonstrates that the proposed abstraction layer does not compromise correctness or compatibility with existing tooling. Instead, it preserves interoperability while offering a different interaction model.

The performance evaluation further complements these findings by highlighting how architectural decisions in \textit{PTERO} translate into measurable workflow improvements. While both systems are capable of executing equivalent \textit{IEC 61499} applications, \textit{PTERO} reduces the number of manual steps required across key development tasks such as project setup, synchronisation and deployment. A central factor behind this improvement is the tighter integration between the different components of the development pipeline. In \textit{4diac IDE}, the development workflow is distributed across multiple loosely connected tools and processes, requiring explicit user actions to maintain consistency between the modelling environment, local file system, and remote runtime. In contrast, \textit{PTERO} internalises these responsibilities within the platform itself, allowing project state, file management, and deployment logic to be handled automatically as part of a single unified system.

The usability questionnaires provide strong evidence that a web-based implementation can offer a significantly improved user experience compared to \textit{4diac IDE}. Participants consistently reported higher levels of perceived ease of use, intuitiveness, and overall satisfaction when interacting with \textit{PTERO}. This improvement can be attributed not only to interface design choices, but also to the inherent advantages of web technologies in this context.

Furthermore, it was observed that the integration of all development tasks into a single unified interface contributes to a reduction in cognitive load. Unlike \textit{4diac IDE}, where users must switch between multiple tools and external processes (such as file system management and synchronisation scripts), the web-based environment consolidates modelling, editing, deployment and monitoring into a coherent workflow. The results also suggest that responsiveness and perceived simplicity play an important role in user satisfaction.

Overall, the findings indicate that the adoption of web technologies is not merely a change in deployment platform, but a design decision that directly influences usability by simplifying access, unifying workflows and reducing structural complexity for end users.
